142 // TODO: It may be possible to make parts of this stream have CRTSFilter
143 // modules running in different processes, not just threads. This should
144 // be able to be added seamlessly at without changing the users CRTSFilter
145 // code.
146 //
147 //
148 //
149 // CRTSFilter trade-offs and considerations:
150 //
151 // If unlocked thread looping process time, for more than one thread in
152 // the stream, is much greater than inter thread contention time per loop,
153 // then we will get better performance with multi-threading. If one
154 // module thread is the bottle-neck (slower than all others) than there
155 // may be no benefit to multi-threading.
156 //
157 // We can put any number of adjacent modules in the same thread.
158 //
159 //
160 // This idea of "stream" seems to fit what we are doing.
161 // https://en.wikipedia.org/wiki/Filter_(computing)
162 //
163 // Base class for crts_radio IO (stream) modules or or call them stream
164 // modules.
165 //
166 // We are also concerned with sources and sinks of these streams.
167 // A source of a stream is a CRTSFilter with no data being written to it.
168 // A sink of a stream is a CRTSFilter with no data being written from it.
169 //
170 // This is a stream like thing using only the exposed input() methods
171 // for each part of the software filter stream.
172 //
173 //
174 // A common filter stream processing interface to seamlessly provide
175 // runtime optimization of threading and process topology selection. Kind
176 // of like in optimization of MPI (massage passing interface) HPC (high
177 // performance computing) applications, we have node/thread runtime
178 // partitioning.
